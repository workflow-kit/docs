    Overview¶

    Rules are organized into 9 categories based on what they detect. Each rule:

    Start small: Enable 3-5 rules that matter most to your team
    Test first: Use debug mode to see how rules behave
    Expand gradually: Add more rules as you see value

    Rule Behavior¶

    Detection Logic¶

    Each rule:

    1. Analyzes PR data ‚Äî Changed files, diffs, metadata
    2. Applies detection patterns ‚Äî File paths, content patterns, PR properties
    3. Returns labels ‚Äî List of labels to apply (can be empty)
    4. Logs activity ‚Äî When debug mode is enabled

    Label Application¶

    • Labels are applied cumulatively ‚Äî multiple rules can add labels
    • Labels are added, not replaced ‚Äî existing labels remain
    • Labels are created automatically ‚Äî if they don't exist in the repo
    • Labels can be customized ‚Äî via label_overrides

    Performance¶

    • All rules run in parallel where possible
    • Average execution time: 2-5 seconds for typical PRs
    • Rules are stateless ‚Äî no side effects or dependencies

    Customization¶

    Override Label Names¶

    Match your team's existing label conventions:

    label_overrides: '{
  "ui-change": "area/frontend",
  "test-missing": "needs-tests",
  "large-pr": "size/XL",
  "security-change": "security-review-required"
}'

    Adjust Thresholds¶

    Customize the large PR threshold:

    large_pr_threshold: '300'  # Default is 500

    Debug Mode¶

    Enable detailed logging to understand rule execution:

    enable_debug: true

    Debug output includes:

    • Files being analyzed
    • Rules being evaluated
    • Pattern matches found
    • Labels being applied
    • Execution time per rule
